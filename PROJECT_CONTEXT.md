# Контекст проекта ВКР

## 1. Тема и постановка

**Тема:** Разработка модели искусственного интеллекта для прогнозирования спроса на перевозку пассажиров автобусами при чрезвычайных событиях на линиях метрополитена

В практической части ВКР планирую построить ML-сервис, который по информации о сбое в метрополитене (какая линия, какое время, предполагаемая длительность) прогнозирует прирост пассажиропотока на ближайших автобусных маршрутах в ближайший час.

Источник данных — открытый датасет Metropolitan Transportation Authority (MTA, Нью-Йорк) с портала data.ny.gov за 2020–2025 гг. Он содержит четыре согласованные таблицы с часовой детализацией:
- пассажиропоток метро по станциям;
- пассажиропоток автобусов по маршрутам;
- журнал крупных инцидентов в метро;
- лог всех оповещений о нарушениях сервиса.

Практическая часть разбивается на три этапа. На первом этапе загружаю и склеиваю четыре таблицы MTA в единый обучающий набор, версионирую через DVC. Выполняю feature engineering — временные признаки (час, день недели, сезон), лаговые (пассажиропоток за предыдущие 15/30/60 минут), топологические (расстояние от автобусной остановки до затронутых станций метро), характеристики самого инцидента. Обучаю три модели для сравнительного анализа: baseline на исторических средних, градиентный бустинг XGBoost с автоматическим подбором гиперпараметров через Optuna, Random Forest. Все эксперименты (ожидается примерно 50 запусков) логируются в MLflow с фиксацией параметров, метрик и артефактов. Сравнение моделей проводится по метрикам MAE, RMSE, MAPE, а также через анализ важности признаков.

На втором этапе оборачиваю лучшую модель в REST API-сервис на FastAPI, упаковываю в Docker-контейнер. Настраиваю мониторинг — логирование входных запросов и выходных предсказаний в SQLite, визуализация работы сервиса через Streamlit-дашборд.

На третьем этапе демонстрирую полный цикл ML-решения на примере двух версий модели: исходная версия и улучшенная после обнаружения в мониторинге расхождения между предсказаниями модели и фактическими значениями на новых данных.

Главная суть — не максимально возможная точность модели, а выстроенный воспроизводимый процесс жизненного цикла ML-решения (связка DVC + MLflow + Optuna + Docker + FastAPI + мониторинг).

Итоговый результат — работающий сервис, доступный по REST API, сравнительная таблица обученных моделей с журналом экспериментов в MLflow, воспроизводимый через Docker код решения.

## 2. План работы (4 главы)

**Введение**

**Глава 1 Анализ предметной области**
- 1.1 Задача прогнозирования транспортного спроса при чрезвычайных событиях (Характеристика предметной области, типология инцидентов, роль автобусного транспорта, обоснование применимости ML и границы его применения)
- 1.2 Обзор методов и существующих решений (классификация подходов: статистические, классический ML, deep learning)
- 1.3 Анализ доступных источников данных
- 1.4 Постановка задачи исследования
- Выводы по главе 1

**Глава 2 Проектирование ML процесса**
- 2.1 Архитектура конвейера жизненного цикла ML (общая схема и обоснование стека DVC, MLflow, Optuna, FastAPI, Docker, Streamlit)
- 2.2 Исходные данные и формирование обучающего датасета
- 2.3 Версионирование данных и инжиниринг признаков (DVC-конвейер и четыре группы признаков с обоснованием)
- 2.4 Выбор моделей и методика автоматизированного обучения (теоретическое обоснование выбора baseline + XGBoost + Random Forest, применение Optuna для подбора гиперпараметров)
- 2.5 Трекинг экспериментов и методика валидации (структура MLflow, метрики качества, time-series split)
- Выводы по главе 2

**Глава 3 Научные эксперименты**
- 3.1 Дизайн и протокол экспериментов (постановка трёх экспериментов, гипотезы, критерии успеха)
- 3.2 Эксперимент 1. Сравнительный анализ моделей (Baseline vs XGBoost vs Random Forest на едином датасете. Метрики, время обучения, время инференса. Скриншоты MLflow)
- 3.3 Эксперимент 2. Ablation study по группам признаков (последовательное исключение групп признаков с замером падения качества, интерпретация)
- 3.4 Эксперимент 3. Влияние гиперпараметров на качество (анализ Optuna trials, распределения лучших значений, карта важности гиперпараметров)
- 3.5 Анализ важности признаков и интерпретация результатов (feature importance XGBoost, объяснение результатов с опорой на литературу)
- 3.6 Систематизация влияющих факторов
- Выводы по главе 3

**Глава 4 Реализация и внедрение решения**
- 4.1 Архитектура внедряемого решения (схема компонентов, диаграмма взаимодействия)

- 4.3 Мониторинг и визуализация (логирование в SQLite, Streamlit-дашборд со скриншотами)

## 8. Дневник проблем и решений

### Day 1 (24.04.2026) — Конфликт версий pathspec и DVC

**Проблема:** При первом запуске `dvc init` получили ошибку
`ImportError: cannot import name '_DIR_MARK' from 'pathspec.patterns.gitwildmatch'`

**Причина:** `pip install` подтянул свежую версию `pathspec==1.1.0`, которая ещё не совместима с DVC 3.56.0. DVC ждёт API старой версии библиотеки.

**Решение:** Откатили pathspec на версию 0.12.x через
`pip install "pathspec<0.13" --force-reinstall`. После этого `dvc init` прошёл чисто. Зафиксировали в requirements-lock.txt для воспроизводимости.

**Вывод:** Свежие версии библиотек — не всегда рабочие. В ML-экосистеме лучше использовать lock-файлы и откатываться на проверенные комбинации при первом же конфликте.

### Day 1 (24.04.2026) — Активация venv из неправильной папки

**Проблема:** Первая попытка `.\.venv\Scripts\Activate.ps1` упала с ошибкой «Имя не распознано как имя командлета».

**Причина:** Команда выполнялась в PowerShell из `C:\Windows\system32`, а venv находится в `C:\metro-bus-ml\.venv`. Относительный путь не разрешился.

**Решение:** Выполнять все команды только в терминале VS Code, который автоматически открывается в папке проекта. После настройки Python Interpreter (`Python: Select Interpreter`) venv активируется автоматически при открытии нового терминала.
